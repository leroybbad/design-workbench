How should annotation state live?

Three architectural approaches for managing annotation lifecycle across screen iterations

A

Client-Side State (helper.js owns everything)

All annotation state lives in the browser — positions, text, lifecycle status. Stored in localStorage keyed by session ID. The events file only gets written when you return to the terminal (batch send). Claude pushes new HTML, helper.js re-renders persisted annotations on top.

Pros

  • Zero server changes — all logic in helper.js
  • Annotations survive page reloads (localStorage)
  • Clean separation: browser owns UI state, events file is just the export format
  • Fast — no round-trip for every note

Cons

  • Re-attaching annotations after Claude pushes new HTML requires matching elements across versions (selectors may shift)
  • localStorage is per-origin, so all sessions on localhost share it (need session-keying)
B

Server-Side State (server.cjs manages annotations)

Annotations are sent to the server via WebSocket immediately. Server writes them to a persistent state/annotations.json file. When Claude pushes a new screen, the server includes existing annotations in the page response. Events file is written from the server's annotation store.

Pros

  • Single source of truth on disk — survives browser crashes, tab closes
  • Server can manage lifecycle transitions (new → addressed) on screen push
  • Multiple browser tabs would see the same annotations

Cons

  • More server complexity — annotation CRUD, state management, embedding in responses
  • WebSocket traffic for every annotation edit (contrary to batch model)
  • Server becomes stateful beyond just file-watching
C

Hybrid — Client owns UI, server owns persistence

Annotations are created and edited client-side (no WebSocket chatter). On "batch send" — when user is done annotating — client writes all annotations to the server via one WebSocket message. Server persists to state/annotations.json and writes the events file for Claude. On new screen push, server sends previous annotations back so client can re-render them.

Pros

  • Fast editing (client-side), durable storage (server-side)
  • Batch send matches the desired workflow
  • Server stays simple — just persist and replay, no CRUD logic

Cons

  • Two places that know about annotation shape (client + server)
  • If browser closes before batch send, unsent annotations are lost

Recommendation

Option A (client-side) is the simplest starting point for a POC. All the annotation logic lives in one file (helper.js), the server barely changes, and localStorage gives you persistence across reloads. The "annotations lost if browser closes" risk from Option C also applies here, but for a design tool where you're actively looking at the browser, it's acceptable. We can evolve toward the hybrid model later if durability matters.
